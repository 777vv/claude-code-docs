# Agent 权限模式

> **本文你将学会：** Codex 的三种权限模式是什么、有什么区别、什么时候该用哪种。

## 为什么需要权限模式？

Codex 是一个能读写文件、执行 Shell 命令的 AI。如果不加限制，它理论上可以：
- 删除文件
- 访问网络
- 修改系统配置

权限模式让你精确控制 AI 的"活动范围"，在效率和安全之间取得平衡。

---

## 三种模式详解

### 🟢 Auto 模式（默认）

**适合：** 日常开发，在自己的项目目录内工作

在 Auto 模式下，Codex 可以**自动**执行以下操作，无需每次确认：

- ✅ 读取当前工作目录内的文件
- ✅ 修改当前工作目录内的文件
- ✅ 在当前目录内执行 Shell 命令

但以下操作**仍需你手动确认**：

- ❓ 访问当前目录之外的文件
- ❓ 发起网络请求
- ❓ 修改系统配置

```bash
# 默认就是 Auto 模式
codex

# 显式指定
codex --approval-mode auto
```

---

### 🟡 只读模式（Read-Only）

**适合：** 代码审查、理解陌生代码库、不想让 AI 改动任何东西

在只读模式下，Codex 只能读文件，**任何修改或命令执行都需要你明确批准**：

- ✅ 读取文件（自动）
- ❓ 修改文件（需确认）
- ❓ 执行命令（需确认）

```bash
codex --approval-mode read-only
```

::: tip 使用场景示例
```
# 在只读模式中用于代码理解
codex --approval-mode read-only

# 然后输入：
帮我梳理整个 src 目录的模块结构，画出依赖关系图
解释 middleware/auth.js 的验证逻辑
```
:::

---

### 🔴 完全访问模式（Full Access）

**适合：** 你明确信任当前仓库，想要 Codex 高度自主地完成复杂任务

在完全访问模式下，Codex 可以跨目录操作、访问网络，几乎不需要确认：

- ✅ 读写任意文件（自动）
- ✅ 执行任意命令（自动）
- ✅ 访问网络（自动）

```bash
codex --approval-mode full
```

::: danger 使用完全访问模式前请注意
1. **只在你完全信任的私有仓库中使用**
2. **确保 git 已初始化**，以便随时回滚
3. **不建议在生产环境服务器上使用**
:::

---

## 模式对比表

| 功能 | Auto | 只读 | 完全访问 |
|------|:----:|:----:|:--------:|
| 读当前目录文件 | 自动 | 自动 | 自动 |
| 写当前目录文件 | 自动 | 需确认 | 自动 |
| 执行 Shell 命令 | 自动 | 需确认 | 自动 |
| 读其他目录文件 | 需确认 | 需确认 | 自动 |
| 网络请求 | 需确认 | 需确认 | 自动 |

---

## 如何在会话中切换模式

你也可以在 Codex 运行时发送指令临时调整行为：

```
# 在对话中告诉 Codex 本次只需要阅读，不要修改文件
这次只需要分析代码，不要做任何修改
```

---

## 在 config.toml 中设置默认模式

如果你每次都想用特定模式，可以写入配置文件：

```toml
# ~/.codex/config.toml
[agent]
approval_mode = "auto"  # 可选值：auto / read-only / full
```

---

## 实际场景推荐

| 场景 | 推荐模式 |
|------|----------|
| 第一次在项目中使用 Codex | 只读，先了解 AI 的行为风格 |
| 日常开发写新功能 | Auto |
| 全量重构复杂模块 | Auto（确保 git clean） |
| 审查同事代码 | 只读 |
| 在受信任的 CI 环境中自动化 | 完全访问 |
| 探索不熟悉的开源项目 | 只读 |
